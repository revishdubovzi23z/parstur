## 0. Что в коде сделано хорошо

Кратко, чтобы не повторяться в каждом пункте:

- **Чистая модульная разбивка.** `routes/`, `runtime/`, `db/`-mixin-ы, `app_core` для общей логики, отдельные клиенты TMDB/Kinopoisk/PoiskKino — границы понятны и стабильны.
- **Settings через Pydantic** (`settings.py`), nested-классы, env-driven. Тестовая фикстура `tests/conftest.py` явно сбрасывает env, проверки на `Field` с `description` — плюс.
- **SQLite приведена в порядок:**
  - WAL + `synchronous=NORMAL`, `busy_timeout=30s`, периодический `wal_checkpoint(TRUNCATE)` на старте и по таймеру (`main.py:81,424`).
  - Кастомный `REGEXP` с `lru_cache` (`db/core.py:19-44`) — приятно и безопасно (whitelisted поля в `FILTER_RULE_FIELDS`, SQL-инъекция через имя колонки невозможна).
  - Партиалные/композитные индексы, FTS5 + диакритик-нечувствительный токенайзер, fallback на LIKE при недоступном FTS.
  - Идемпотентные миграции (`PRAGMA user_version`), материализация больших `IN (...)` в `TEMP TABLE` (`db/core.py:67-81`).
- **Сессии в БД, а не в памяти процесса** (`db/sessions.py`). Хэширование токена SHA-256, sliding TTL, GC-loop. Это решает «после рестарта всех логаут» из ранних версий.
- **CSP без `unsafe-eval`** (`main.py:315`), `X-Content-Type-Options: nosniff`, `frame-ancestors 'none'`, `Permissions-Policy`. Для Vue 3 в режиме compile-time это нормальная политика.
- **WebSocket аутентификация** через одноразовые тикеты (`/ws?ticket=...`) — лучше, чем `?token=` в логах reverse-proxy.
- **Lifespan-менеджер** заменил `@on_event` — корректно cancel-ит фоновые задачи и делает финальный stop-flag перед `terminate()` (`main.py:56-134`).
- **Health-эндпоинт** возвращает `503` при недоступной БД (`main.py:266-296`), `HEALTHCHECK` в Dockerfile, non-root user `par2:1000`.
- **`audit_log` + `/audit_log/{id}/undo`** — приятная защита от случайных правок.
- **Backup/restore с magic-bytes валидацией** (`routes/admin.py`), confirmation-токен `/api/reset_database` с 60-сек TTL.
- **`api_cache.db`** через `CachedSession` — экономит лимиты TMDB/Kinopoisk.
- **Тестовая база разумная:** ~30 файлов в `tests/`, в т.ч. `test_auth_persistence.py`, `test_rezka_relogin.py`, `test_migrations.py`, `test_collections_export_import.py`.
- **CI с матрицей шагов**: lint, types, tests, docker build, frontend build (`.github/workflows/ci.yml`).
- **Фронт чист от XSS-векторов** — нет ни одного `v-html` / `innerHTML` / `eval` (`grep` подтверждает).
- **`AGENTS.md` + `ROADMAP.md`** — редкая для домашних проектов вещь, реально помогает разговаривать с агентом.

В целом для «личного трекера» это качество близкое к продуктовому. Дальше — про то, что я бы поправил.

## 1. Критические и средне-критические находки

### 1.1. Stop-флаг пишется не туда, куда читается (Bug)

**Симптом.** Кнопка «Стоп» в SyncPanel не останавливает фоновую задачу при стандартном Docker-деплое (`WORKDIR=/app`, `APP_DATA_DIR=/data`). Запрос успешно возвращается, но скрипт продолжает идти до конца категории.

**Где:**
- `routes/process.py:148`, `:161` — пишет `open(f"stop_{key}.flag", "w")` относительно CWD.
- `script_utils.py:38,93` — `should_stop()` / `clear_stop_flag()` читают из `APP_DATA_DIR` (`_get_data_path`).
- `main.py:123` (shutdown) — пишет уже корректно через `os.path.join(settings.app_data_dir, ...)`.

То есть shutdown работает, а ручное «Стоп» — нет (для не-CWD-инсталляций). Тривиальный one-line bug, но не покрыт тестом.

**Рекомендация.** Завести единый хелпер `runtime.stop_flag.path(key)` и звать его и из `routes/process.py`, и из `main.py`, и из `script_utils.py`. Параллельно добавить unit-тест, который монтирует временный `APP_DATA_DIR` и проверяет, что `set_stop()` создаёт файл там же, где `should_stop()` его ищет.

### 1.2. `/api/self_update` — слишком многое на одной кнопке (Risk)

`routes/admin.py` принимает POST и делает `git pull`, `pip install -r requirements.txt`, `cd frontend && npm install && npm run build`, затем триггерит рестарт. Это нормально для homelab, но:

- **Нет confirmation-токена / двухшаговости** — в отличие от `/api/reset_database` (там одноразовый токен на 60с).
- **Нет whitelist origin/branch.** Git pull выполнится из любой ветки, которую кто-то локально переключил.
- **`subprocess.run` с `timeout=60`** на `npm install` — на «холодной» машине этого почти всегда не хватит, и установка прервётся посреди обновления зависимостей, оставив `node_modules` в полуразобранном состоянии.
- **Любой токен с правами админа = RCE.** Сессии переживают рестарты (по дизайну), значит украденный токен один раз == постоянный доступ к деплою.

**Рекомендация.**
1. Сделать confirmation-токен как у reset (двухстейджевый).
2. Хранить `EXPECTED_REMOTE_URL` и `EXPECTED_BRANCH` в settings; перед `git pull` сверять `git remote get-url origin` и `git rev-parse --abbrev-ref HEAD`.
3. Поднять timeout npm-шага до 10 мин или вынести его в отдельный фоновой процесс с прогресс-репортом (через тот же `process_status`).
4. Логировать short-SHA до и после pull в audit_log.
5. Рассмотреть «soft-disable» self-update флагом `ENABLE_SELF_UPDATE` (default false).

### 1.3. `/api/database_import` — без проверки схемы

Magic-bytes (`SQLite format 3\x00`) проверяются, но импортированная БД может быть от любой версии схемы. После рестарта `init_schema` накатит миграции, но если в импортированной БД есть таблицы из будущей версии или конфликтующая `user_version`, можно получить неработающий бутстрап.

**Рекомендация.** Перед заменой основной БД:
1. Открыть импортированный файл readonly.
2. Прочитать `PRAGMA user_version` и сверить с `db.MIGRATIONS[-1]`.
3. Прогнать `PRAGMA integrity_check` и `quick_check`.
4. Запустить `init_schema` на копии, чтобы убедиться, что миграции пройдут.
Только потом — `os.replace` и рестарт.

### 1.4. `single_update` не обозначен в `check_any_running`

`routes/items.py:60-67` ставит задачу в `task_queue`. Очередь однопоточная, так что данных не покалечит. Но:

- Юзер кликает «Обновить элемент» во время идущего sync — получает `started: true` и через 5 минут «магически» появляется обновление. Никакой обратной связи о том, что задача в очереди.
- В SyncPanel/AdminPanel UI продолжает показывать «не запущено» для single_update, потому что process_status для него не обновляется как для других.

**Рекомендация.** Либо вызывать `check_any_running` и возвращать `409` с указанием активной задачи, либо честно добавить single_update в process_status / в WS-broadcast о «pending в очереди» с указанием позиции.

### 1.5. Rate-limit только на login и только по `get_remote_address`

`Limiter(key_func=get_remote_address)` берёт `request.client.host`, который за reverse-proxy всегда равен `127.0.0.1` (или IP прокси). Это:

- ломает rate-limit полностью при типичном деплое (`nginx → uvicorn`);
- наоборот, может ложно «запереть» нескольких легитимных клиентов одного офиса.

Кроме того, лимита нет ни на `/api/database_import`, ни на `/api/self_update`, ни на `/api/reset_database/token`, ни на `start_*`/`stop_*`. Все они принимают только заголовок `Authorization`, но если кто-то получит токен (или auth выключен), он сможет генерировать рестарты пачкой.

**Рекомендация.**
1. Включить поддержку `X-Forwarded-For`/`X-Real-IP` через настройку `forwarded_allow_ips` в uvicorn и кастомный `key_func`, который берёт первый IP из XFF при доверенных прокси.
2. Добавить умеренные лимиты на «опасные» админ-эндпоинты (`5/hour`).
3. Удалить fallback на `AUTH_PASS` в plain-text или логировать `WARNING` при старте.

### 1.6. Синхронный `time.sleep` в HTTP-обработчиках

`base_client._handle_rate_limit` спит `time.sleep(5)` (`base_client.py:74`). Тот же `BaseMovieClient` используется в `routes/streams.py` для трейлеров и в TMDB-вызовах из эндпоинтов. Если TMDB вернёт `429` — обработчик заблокирует поток пула FastAPI на 5 секунд. Sync-скриптам это окей (они в отдельном subprocess), но фон-обработчикам — нет.

Аналогично `routes/streams.py` делает сетевые запросы к `kinobox.tv` / `fbphdplay.top` напрямую `requests.get(...)` внутри `async def` — это блокирует event-loop:

```python
# routes/streams.py
async def online_sources(...):
    r = requests.get("https://kinobox.tv/...", ...)   # блокирует loop
```

**Рекомендация.**
1. Все сетевые I/O из `async def`-обработчиков — через `httpx.AsyncClient` или `await run_in_threadpool(requests.get, ...)`.
2. В `base_client` заменить `time.sleep` на принимающий «sleep_fn», а в async-режиме передавать `asyncio.sleep`.
3. Лучше отметить такие тяжёлые эндпоинты `def` (а не `async def`) — FastAPI сам отправит их в threadpool и не будет блокировать loop.

## 2. Архитектурные замечания

### 2.1. Смешение sync/async обработчиков и прямые SQLite-вызовы

В большинстве `routes/*.py` обработчики — `async def`, но вызовы `db.<метод>()` синхронные. Пока нагрузка маленькая, это нормально, но:

- При параллельных WS-broadcast (gather по подключённым клиентам) и одновременном HTTP-запросе блокирующая SQLite-операция «съест» весь рабочий loop.
- `task_queue` ставит фоновые задачи в очередь, но HTTP-роут, который их инициирует, всё равно делает SQL-запросы рядом.

**Рекомендация.** Принять один из двух стилей и придерживаться его:
- либо обернуть DB-зов в `await run_in_threadpool(db.x, ...)` для всех `async def`;
- либо сделать обработчики `def` (Starlette сам отправит в threadpool).

### 2.2. `db.get_connection()` вне контекст-менеджера

Несколько мест (например, `routes/streams.py`, `runtime/rezka.py`) делают:
```python
conn = db.get_connection()
conn.execute(...)
conn.commit()
conn.close()
```
вместо штатного `with db._conn() as c:`. Это:
- ломает гарантии `_conn()` (rollback, прокидывание PRAGMA, REGEXP функция);
- при исключении между `execute` и `close` оставляет файловый дескриптор / WAL в подвешенном состоянии.

**Рекомендация.** Сделать `db._conn` единственной публичной точкой получения соединения; пометить `db.get_connection` deprecated и грепнуть кодбазу.

### 2.3. Магические `category_id` и form-объекты feed-фильтров

`db/items.py:get_feed` — функция на 200+ строк с десятком ветвей по `category_id` (`-1` = все, `-2` = ignored, `-100..-104` = pseudo-категории по году/рейтингу). Конкретно:
- логика «спрятать просмотренное» (`hide_collected`) дублируется в трёх местах;
- параметры (`page`, `limit`, фильтры) проверяются вручную через `int(...)`-конверсии в роуте;
- `latest_release` высчитывается коррелированным подзапросом, потом снова батчем для items — два прохода по `releases`.

**Рекомендация.**
1. Ввести `FeedFilter` Pydantic-модель (валидация на границе) и `FeedSort` enum.
2. Заменить магические category-id на `enum FeedCategory`.
3. Выделить query builder (`feed_query.py`), оставив в `get_feed` только композицию.
4. Подсчёт `latest_release` сделать одним JOIN-ом `releases r ON r.item_id = items.id AND r.date_added = (SELECT MAX(...))` или через оконные функции (SQLite ≥3.25 поддерживает).

### 2.4. Несовместимая «форма» ответов

В одном и том же коде встречается:
- `return {"status": "success"}`
- `return {"id": rid}`
- `return JSONResponse({"error": str(e)}, status_code=400)`
- `return Response("Unauthorized", status_code=401)`
- `raise HTTPException(...)`

Фронт-стор должен помнить, какой эндпоинт что возвращает. Это уже приводит к коду в `frontend/src/stores/admin.ts`, который ловит и `res.json().error`, и `res.text()` — оба варианта работают, но это шум.

**Рекомендация.**
1. Единый `Envelope`-протокол: всегда `{"ok": bool, "data": ..., "error": ...}` (или RFC-7807 problem+json).
2. Перевести фронт на одну функцию-парсер.
3. Для ошибок — `HTTPException` + кастомный exception_handler, который сериализует в этот же формат.

### 2.5. Дублирование стоп-флагов и «pipeline_stop_requested»

Состояние «пользователь нажал стоп» сейчас живёт **в трёх местах**:
- модуль-глобал `runtime.processes.pipeline_stop_requested`;
- файлы `stop_<key>.flag` на диске;
- статус в `process_status[key] = "stopped"`.

Они синхронизуются вручную. Это потенциальный источник «зомби-задач»: pipeline стопится, но единичный sync_video в очереди уже стартует следующим, потому что `pipeline_stop_requested` дёрнулся обратно в False между шагами.

**Рекомендация.** Завести `class Stopper:` с методами `request_stop(key)`, `is_stopped(key)`, `clear(key)` и единым диском под капотом. Удалить модуль-глобал.

### 2.6. Hard-coded внешние сервисы

`kinobox.tv`, `fbphdplay.top` (стримы), `https://rezka.ag` (HDRezka), `https://rutor.info` (Rutor) зашиты в код. В `.env.example` упомянуты только TMDB/Kinopoisk.

**Рекомендация.** Вынести URL в settings (`ONLINE_SOURCE_KINOBOX_URL`, `RUTOR_BASE_URL`, `REZKA_BASE_URL`) с дефолтами текущими — это и про configurability, и про дисклеймер «эти хосты могут менять домен».

### 2.7. Хардкод сид-коллекций

`db/core.py:344-355` создаёт дефолтные коллекции с русскими названиями (включая «говноозвучки» — это, наверное, шутка, но в публичном репо смотрится резко). Эти строки попадают в SQL-инициализатор схемы.

**Рекомендация.** Перенести сидинг в отдельный `seeds/default_collections.json`, читать при первом запуске. Это и кода чище, и можно дать пользователю заменить дефолты, не правя миграции.

## 3. Безопасность

(Сверх того, что попало в раздел «критические».)

### 3.1. CSP `style-src 'unsafe-inline'`

Нужен для `:style="..."` в Vue. ОК на сегодня; в долгосрочной перспективе можно переехать на CSS custom properties + классы и закрыть его.

### 3.2. `connect-src ... https:`

Очень широко (любой https-ресурс). Если посадить параноидальную CSP, можно сократить до конкретного списка (`https://api.themoviedb.org`, `https://kinopoiskapiunofficial.tech`, `wss://example.com`).

### 3.3. Логи и токены

Авторизационный токен в `?token=...` бывает в access-логах reverse-proxy. Тикеты решают эту проблему для WS, но `/api/...` всё ещё разрешает токен в URL? — нет, не разрешает, только в заголовке. Хорошо. Но проверить, что нигде не логируется `request.headers["Authorization"]` (быстрый grep — не логируется).

### 3.4. `subtitle_proxy` (если есть) и SSRF

В `routes/streams.py` упоминается `_SUBTITLE_HOST_ALLOWLIST` — это правильный паттерн. Стоит распространить тот же подход на любые «прокачивающие чужой контент» эндпоинты (m3u-proxy, online sources).

### 3.5. `INSERT OR REPLACE` на сессиях

При коллизии токена `INSERT OR REPLACE` уничтожит «старую» сессию — на практике невозможно (`secrets.token_hex(32)` = 256 бит), но если в проекте появятся session-attached данные (history, preferences) — каскад REPLACE их сотрёт. Лучше явный `INSERT … ON CONFLICT DO UPDATE`.

## 4. Производительность и устойчивость

### 4.1. `requests` без пула соединений

В `rutor_parser.py`, `single_item_update.py`, `runtime/rezka.py` — на каждый вызов поднимается новый TCP/TLS-сокет. На скрейпе сотен тайтлов это заметно.

**Рекомендация.** Использовать `requests.Session()` (или `httpx.Client`) в `BaseMovieClient` и пробрасывать в наследников. У `RutorParser` это особенно дёшево сделать.

### 4.2. Trailer/online-sources в request-thread

См. 1.6 — `routes/streams.py` блокирует пул FastAPI на 1-2 сек на каждый клик «трейлер». При нескольких клиентах одновременно — ощутимо.

### 4.3. `releases` подзапросы

В `get_feed` используется `(SELECT MAX(date_added) FROM releases WHERE item_id = items.id)` как correlated subquery — окей для текущего объёма, но если `releases` вырастет до десятков тысяч строк, лучше держать денормализованную колонку `items.latest_release_at` и обновлять триггером на `releases`.

### 4.4. `_handle_rate_limit` обрабатывает 401/403 как rate-limit

В `kinopoisk_client`/`poiskkino_client` `rate_limit_codes` включает `401, 403`. То есть при неверном API-ключе клиент будет «спать и ретраить» вместо того, чтобы быстро упасть с понятной ошибкой.

**Рекомендация.** Разделить: 401/403 → `AuthError` (фатальная), 429/503 → retry.

### 4.5. `run_command` интерпретирует exit codes

`runtime/processes.py:344` — `elif proc.returncode in (-15, 1, 15): status = "stopped"`. Засчитывать `1` как «остановлено» опасно: любой Python-скрипт упавший с обычной ошибкой получит `returncode=1`. В UI это станет «Остановлено пользователем», и человек не пойдёт смотреть лог.

**Рекомендация.** Скрипты, поддерживающие graceful stop, должны явно возвращать `2` (или `99`) при обнаружении stop-flag, а остальные exit codes сводить к `failed`.

### 4.6. WS broadcast — без backpressure

`runtime/ws.py` рассылает `asyncio.gather`-ом всем клиентам. Если хотя бы один клиент медленный, корутины повисают на `send_json`. На фронте это не страшно, но при отвалившейся вкладке без `unload` сервер будет ждать TCP-timeout.

**Рекомендация.** Обернуть каждую отправку в `asyncio.wait_for(..., timeout=2)` и при таймауте — `disconnect(ws)`.

## 5. Логика и UX

### 5.1. Поведение «авто-логаута»

При 401 `apiFetch` бросает `UnauthorizedError`, в `session.ts` это переводит стор в `unauthenticated`. Но многие сторы (`items.ts`, `feed.ts`) делают `apiFetch(...).then(r => r.json())` без `try/catch` — необработанная ошибка пишет «An unhandled error» в консоль.

**Рекомендация.** Завести `apiJson<T>()` хелпер, который ловит `UnauthorizedError`, переводит стор и возвращает «безопасный» `null`. Альтернатива — глобальный `app.config.errorHandler` в `main.ts`.

### 5.2. Service Worker и кэш фронта

SW кэширует `frontend/dist/assets/*` через precache list, инвалидируется хэшом `_sw_version()`. Это хорошо. Но в `manifest.json` и `sw.js` не описана стратегия для `/api/*` — там по умолчанию идёт network-only. Хорошо, потому что иначе кэшировали бы авторизованные ответы.

Но: при возврате 401 SW может отдать precached `index.html` оффлайн — нужно проверить, что после логаута фронт **запросит у сервера** свежую страницу, а не возьмёт из cache (`navigation preload` + `Cache-Control: no-cache` для `/` уже стоит — окей).

### 5.3. Single-update — флоу без обратной связи

См. 1.4. Дополнительно: фронтовый `items.update(item)` сейчас сразу делает оптимистическое обновление; если фон упал, состояние не откатится. Нужен либо WS-эвент `single_update_done`, либо опрос `/api/items/{id}`.

### 5.4. Кнопки `start_*` без подтверждения

Большие операции (`start_rezka`, `start_reprocess`, `start_cleanup`) запускаются по клику без диалога. Учитывая что они могут идти часы — стоит маленький confirm-диалог либо «hold to confirm» (2-сек удержание).

### 5.5. `_LOG_FILES` ↔ `status_key` маппинг

В `routes/process.py` `_LOG_FILES["fix_poiskkino"]` соответствует `process_status["poiskkino"]`. Для других ключей совпадает напрямую. Это путает: при чтении лога через `/api/sync_log/{key}` пользователь ожидает тот же ключ, что и в `process_status`. Стоит выровнять.

### 5.6. Названия логгеров

`logging.getLogger("parsclode.db")` и т.п. — `parsclode` — рудимент. Сейчас название продукта `par2` / Antigravity Tracker.

**Рекомендация.** Переименовать в `par2.db`, `par2.runtime`, `par2.routes.feed`. Это и красивее в логах, и облегчает фильтрацию.

### 5.7. Двойной `logger = logging.getLogger(...)` в `db/core.py`

Литералы строк 10 и 12 — буквально дубликат. Не баг, но шум.

## 6. Тестирование

### 6.1. Что покрыто

`tests/`:
- авторизация (persistence, relogin),
- миграции,
- save-checkpoint, search names, find_existing_item,
- импорт/экспорт коллекций,
- backup, health, SPA mount,
- settings, logging-config, db helpers.

### 6.2. Чего не хватает (по убыванию приоритета)

1. **Stop-flag E2E**: запускаем «фейковый» sync, нажимаем `/api/stop/{key}`, проверяем что флаг попал туда, куда читает скрипт.
2. **Self-update гард**: smoke-тест, что без confirmation-токена `/api/self_update` отвечает 400 (когда внедрим рекомендацию из 1.2).
3. **DB-import**: подсунуть валидный SQLite, но с несовместимой `user_version`, проверить что 400.
4. **Feed-фильтры**: `get_feed(category_id=-2)` — игнорированные с пагинацией; `min_kp`/`max_kp` границы.
5. **Rate-limit за прокси**: проверка что `X-Forwarded-For` учитывается, а не доверяется слепо.
6. **WS broadcast timeout**: тест, что медленный клиент не блокирует остальных.
7. **API envelope**: unit-проверки, что все эндпоинты возвращают одну форму (когда внедрим 2.4).
8. **Streams endpoints**: моки на kinobox/fbphdplay/HDRezka и тесты на graceful-degradation.

### 6.3. Coverage

`pytest --cov` не настроен (по крайней мере, не виден в `pyproject.toml` явно). Минимальный шаг — добавить `coverage.xml` артефакт в CI и порог `--fail-under=70` для backend и `vitest run --coverage` для фронта.

## 8. Конкретный план-минимум на ближайший PR-набор

Если выбирать из всего вышеперечисленного «топ-5 PR на неделю»:

1. **PR-A: фикс stop-flag** (`runtime.stop_flag` хелпер + два callsite + unit-тест). 30 минут.
2. **PR-B: self-update двухшаговый + audit_log** (по образцу `/api/reset_database`). 1-2 часа.
3. **PR-C: `httpx.AsyncClient`-обёртка для внешних HTTP** (заменить `requests` в `routes/streams.py` и в `BaseMovieClient` при использовании из роутов). 3-4 часа.
4. **PR-D: единая форма API-ответов** (Envelope + апдейт фронт-сторов). 1 день (много мест).
5. **PR-E: `Stopper`-класс + удаление модуль-глобалов** (`pipeline_stop_requested`). 2-3 часа.

Эти пять полностью закроют все «красные» риски, выровняют API-контракт и снимут самые заметные DX-боли. Архитектурные «выпрямления» (2.3 рефакторинг `get_feed`, 4.3 денормализация `releases`) — отдельный спринт.

## 9. Что мне субъективно понравилось

- Подход «один stage = одна PR» и комментарии вида `# ROADMAP Stage 10.7z — ...` прямо в коде — очень помогают читать и понимать «почему так, а не иначе». Сохранить.
- Использование `lru_cache` для скомпилированных регексов в фильтрах — маленький, но правильный жест.
- Sliding TTL для сессий с GC-loop, тикеты для WS, hashed-storage — заметно лучше, чем «обычный» Bearer/in-memory-set.
- Migration runner с idempotent guard на повторных `ADD COLUMN` (через try/except в коде миграций) — экономит часы на расследование «почему боот не идёт после rollback».
- Подробные `logger.info("[STAGE] ...")`-ярлыки в логах — облегчают `grep` в проде.

## 10. Резюме

Проект **готов к продакшну для своего домашнего сценария**. Это уже не «pet hack», это аккуратный mini-framework вокруг SQLite + FastAPI. Найденные пункты делятся на три ясные группы:

- **Багфиксы** (stop-флаг, single_update queue, exit-code mapping) — мелкие, без них вечные «странности».
- **Безопасность админ-эндпоинтов** (self_update, db_import, rate-limit за прокси) — обязательны до того, как сервис будет открыт наружу даже за basic-auth.
- **Архитектурные улучшения** (envelope API, async I/O, Stopper, рефакторинг `get_feed`) — на год вперёд; делают код заметно проще к развитию (Stage 11–12 из ROADMAP идут как раз туда).

Ничего из этого не требует переписывать «вокзал» — каждый пункт изолирован и ложится в формат «1–2 маленьких PR», как и обещано в ROADMAP.

— конец отчёта.
