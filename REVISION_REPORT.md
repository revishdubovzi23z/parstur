## Что сделано хорошо

- **Graceful degradation.** `cloud_sync` и `proxy_manager` используют ленивый импорт (`libsql`, `PySocks`/`proxy_manager`) и не падают, если опциональная зависимость не установлена — приложение поднимается без них.
- **Безопасность секретов.** `.gitignore` закрывает `.env`, `*.pem`, `*.db`, `*_log.txt`, `venv/`, `node_modules/`. `CLOUD_TURSO_TOKEN` добавлен в `SENSITIVE_CREDENTIAL_KEYS`, т.е. не возвращается наружу в `/api/settings/credentials`.
- **Миграция через user_version.** `0009_cloud_sync_log.sql` подключается общим раннером, гейтится на `PRAGMA user_version`, идемпотентна.
- **Опциональные зависимости закомментированы** в `requirements.in` — дефолтный lock остаётся тонким, фичи включаются осознанно.
- **Документация в `.env.example`** подробная и честная (включая предупреждения про vless/Xray и PySocks).

## Находки и рекомендации

### 🟡 1. TMDB проксируется только частично
`tmdb_client.py`: прокси (`_get_proxies()`) добавлен в 4 метода метаданных — `get_external_ids`, `get_videos`, `find_by_imdb_id`, `search_movie`. Но методы TMDB **v4-списков** (`get_user_lists`, `create_list`, `update_list`, `delete_list`, `get_list_items`, `add_items_to_list`, `remove_items_from_list`, `is_list_alive`) ходят на `api.themoviedb.org` **напрямую, без прокси**.

Если цель Stage 14 — обход геоблокировки/стабильность, то v4-вызовы при заблокированном доступе будут падать, хотя поиск/метаданные работают через прокси — поведение получится непоследовательным.

**Рекомендация:** централизовать проксирование. Например, обернуть `self.session` методом-хелпером `self._get(url, **kw)`, который сам подставляет `proxies=self._get_proxies()`, и перевести все вызовы `self.session.get/post/put/delete` на него. Это уберёт дублирование и закроет все эндпоинты разом.

### 🟡 2. Нет тестов на новый код (Stage 13/14)
331 тест не покрывает `cloud_sync.py`, `proxy_manager.py`, `routes/cloud.py`, `routes/proxy.py`.

**Рекомендация:** добавить юнит-тесты хотя бы на чистую логику (без сети):
- `proxy_manager.get_requests_proxies()` — парсинг `http://`/`socks5://` (→ `socks5h`), пустое значение → `None`, неизвестный сервис.
- `cloud_sync.enabled` — гейтинг на `provider=="turso"` + наличие url/token.
- `cloud_sync.get_status()` — форма ответа без реального libsql (мок).
- Роуты `/api/cloud/status` и `/api/proxy/status` через `fastapi.testclient` (httpx уже есть в деве).

### 🟡 3. Риск несогласованного снимка при Cloud `push()` / потеря данных при `pull()`
- `push()` зеркалит всю БД построчно. Если он стартует во время активной записи (`sync_job`, фоновые задачи), возможен **несогласованный снимок** (часть таблиц — до изменения, часть — после).
- `pull()` на старте перезаписывает локальную БД из облака. Если облако устарело (например, push не прошёл в прошлый раз), есть **риск отката локальных изменений**.

**Рекомендация:**
- Снимать push из консистентного источника: `VACUUM INTO` во временный файл или SQLite backup API под блокировкой, и заливать его, а не читать живую БД построчно.
- В `pull()` сверять направление по `PRAGMA user_version` / метке времени из `cloud_sync_log` и логировать, что именно перезаписывается; при подозрении на регресс — делать локальный бэкап перед перезаписью (бэкап-механизм в проекте уже есть — `db.backup_to`).

### 🟡 4. Рассинхрон конфигурации pytest (`asyncio_mode`)
В прогоне: `plugins: anyio-4.13.0` и предупреждение `PytestConfigWarning: Unknown config option: asyncio_mode`. То есть в `pyproject.toml` задан `asyncio_mode = "auto"` (опция **pytest-asyncio**), но сам плагин `pytest-asyncio` не установлен — работает только `anyio`.

Это значит: либо опция лишняя, либо async-тесты исполняются не так, как задумано.

**Рекомендация:** определиться — если используем `anyio`, убрать `asyncio_mode` из `[tool.pytest.ini_options]`; если нужен pytest-asyncio, добавить его в `requirements-dev.in`. Сейчас спасает только то, что это warning, а не error (несмотря на `--strict-config`).

### 🟡 5. «Мёртвый» тест в корне — `test_tmdb_lists.py`
Файл лежит в корне проекта, а `testpaths = ["tests"]`. Значит он **не собирается и не запускается** pytest — ложное ощущение покрытия.

**Рекомендация:** перенести в `tests/` (если актуален) или удалить (если устарел).

### 🟢 6. `_get_proxies()` пересчитывается на каждый запрос
В `base_client` и `tmdb_client` `_get_proxies()` делает `from proxy_manager import proxy_manager` + lookup на каждый HTTP-вызов. Дёшево, но избыточно.

**Рекомендация:** кешировать результат на уровне клиента и инвалидировать при `reload_settings()` / `/api/proxy/reload`.

### 🟢 7. Xray/vless без супервизора
`ensure_xray_running()` спавнит процесс, но нет перезапуска при падении и полноценного health-check готовности SOCKS-порта (кроме базовой проверки).

**Рекомендация:** для прод-сценария добавить проверку живости порта с backoff и авто-перезапуск, плюс корректную остановку — `stop_all()` уже вызывается на shutdown, это хорошо.

### 🟢 8. Гигиена корня репозитория
В корне много одноразовых скриптов и мусора: `fix_posters.py`, `cleanup_duplicates.py`, `single_item_update.py`, `reprocess_database.py`, `preview_csv.py`, `scratch_dump.txt`, `scratch_title.txt`, каталог `scratch/`.

**Рекомендация:** вынести служебные скрипты в `scripts/` (или `tools/`), а `scratch/` и `scratch_*.txt` добавить в `.gitignore`. Это упростит навигацию и снизит шум для новых контрибьюторов.

### 🟢 9. CSV вне `.gitignore`
`backup_1379576_votes.csv` в корне **не** попадает под существующие паттерны `.gitignore` (`IMDBOCENKI.csv`, `kinopoiskocenki.csv` перечислены явно, а этот — нет) и может случайно уйти в коммит.

**Рекомендация:** добавить `*.csv` или конкретный файл в `.gitignore` (с учётом, что нужные шаблоны данных, если такие есть, надо исключить из правила).

### 🟢 10. mypy покрывает только `script_utils`
Новый код (`cloud_sync`, `proxy_manager`) без типовой проверки в CI.

**Рекомендация:** по мере типизации добавлять модули в `[[tool.mypy.overrides]]` — начать с `proxy_manager` (там простые типы).
